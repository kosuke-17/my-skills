# Phase 6: Feature コンポーネント実装 — 詳細手順

feature ディレクトリ内の各コンポーネントを、依存関係の順に実装する。

---

## Step 6-1: feature コンポーネントの実装順序の決定

### やること

- Phase 2 のファイルマップから全 feature コンポーネントを列挙する
- 依存関係を分析し、実装順序を決定する

### 順序決定の原則

```
1. 末端（Leaf）コンポーネントを先に実装する
   - 他のコンポーネントに依存しないもの
   - props を受け取って表示するだけのもの

2. 中間（Container / Composite）コンポーネントを後に実装する
   - 子コンポーネントを組み合わせるもの
   - 状態管理を持つもの

3. 最上位コンポーネントを最後に実装する
   - feature のエントリーポイント
   - page.tsx から直接使われるもの
```

### 出力フォーマット

```
## 実装順序

### Layer 1（末端・依存なし）
1. user-table-row.tsx — テーブルの1行表示
2. user-table-header.tsx — ソート付きヘッダーセル
3. user-page-header.tsx — ページタイトル + ボタン

### Layer 2（Layer 1 に依存）
4. user-table.tsx — テーブル本体（row + header を組み合わせ）
5. user-filter-bar.tsx — フィルター入力群

### Layer 3（Layer 1-2 に依存）
6. user-create-modal.tsx — フォーム付きモーダル

### Layer 4（最上位）
（page.tsx は Phase 7 で実装）
```

### 完了条件

- 全コンポーネントの実装順序が決定している
- 依存関係が可視化されている

---

## Step 6-2: 末端コンポーネント（Leaf）から実装開始

### やること

- Layer 1 のコンポーネントを順番に実装する
- 各コンポーネントは Step 6-3 の手順に従う

### Leaf コンポーネントの特徴

```
- props を受け取って表示するのが主な責務
- 自身で状態を持たない（or 最小限のローカル状態のみ）
- イベントはコールバック props で親に伝播する
- テストしやすい
```

---

## Step 6-3: 各コンポーネントの実装（サブステップ）

**全コンポーネントに対してこの手順を繰り返す。**

### Step 6-3a: JSX構造の実装

#### やること

- コンポーネントのJSX構造を組み立てる
- まだスタイリングはしない（構造のみ）

#### 手順

```
1. 'use client' ディレクティブの要否を確認（Phase 2 の SC/CC 分類参照）
2. Phase 3 の Props 型を import する
3. 関数コンポーネントの骨格を作成する
4. Props を分割代入で受け取る
5. セマンティックHTML要素を選択する（div の濫用を避ける）
6. 条件分岐レンダリング（状態パターン別）を実装する
7. key 属性の設定（リスト要素）
8. children / render props の受け取り（該当する場合）
```

#### チェックリスト

```
□ セマンティックHTML（section, article, nav, header, main, aside, footer 等）
□ 見出しレベルの適切さ（h1 → h2 → h3 の順序）
□ リスト要素の適切な使用（ul/ol + li）
□ フォーム要素の label 関連付け
□ 画像の alt 属性
□ key 属性（map で生成する要素）
```

### Step 6-3b: スタイリングの適用

#### やること

- プロジェクトのスタイリング方式に合わせてスタイルを適用する

#### スタイリング方式の確認

```
□ Tailwind CSS → className に utility classes
□ CSS Modules → import styles from './component.module.css'
□ styled-components / Emotion → styled() / css``
□ vanilla CSS → className + 外部CSSファイル
□ インラインスタイル → style={{}}
```

#### チェックリスト

```
□ レスポンシブ対応（Phase 0 で定義した範囲）
□ ダークモード対応（必要な場合）
□ ホバー / フォーカス / アクティブ状態のスタイル
□ disabled 状態のスタイル
□ トランジション / アニメーション（控えめに）
□ スペーシングの一貫性（デザインシステムのスペーシングスケール使用）
□ カラーの一貫性（デザイントークン / CSS変数使用）
```

### Step 6-3c: イベントハンドラの接続

#### やること

- Phase 1 のインタラクション一覧に基づいてイベントハンドラを実装する

#### 手順

```
1. インタラクション一覧から、このコンポーネントに関連するイベントを抽出する
2. コールバック props（onXxx）経由で親に伝播するハンドラを実装する
3. ローカルで完結するハンドラ（ホバー表示切替等）を実装する
4. イベントオブジェクトの型を明示する
5. preventDefault / stopPropagation の必要箇所を確認する
```

#### コーディング規約

```typescript
// ハンドラの命名: handle + 動詞 + 対象
const handleRowClick = useCallback(
  (userId: string) => {
    onRowClick(userId)
  },
  [onRowClick]
)

const handleSortChange = useCallback(
  (key: string) => {
    onSortChange({
      key,
      direction:
        sortConfig.key === key && sortConfig.direction === 'asc'
          ? 'desc'
          : 'asc',
    })
  },
  [sortConfig, onSortChange]
)

// フォームのサブミット
const handleSubmit = useCallback(
  async (e: React.FormEvent) => {
    e.preventDefault()
    // ...
  },
  [
    /* deps */
  ]
)
```

#### チェックリスト

```
□ 全インタラクションに対応するハンドラがある
□ useCallback でメモ化されている（必要な場合）
□ イベントの型が明示されている
□ エラー時のユーザーフィードバックがある
```

### Step 6-3d: 状態管理ロジックの組み込み

#### やること

- このコンポーネントが持つローカル状態を実装する
- Phase 4 のカスタムフックと接続する

#### 手順

```
1. このコンポーネントが持つべきローカル状態を特定する
   - モーダル開閉 → useState<boolean>
   - フォーム入力 → useState or カスタムフック
   - アニメーション状態 → useState

2. 親から受け取る状態と、自身で管理する状態を分離する
   - 制御コンポーネント: 親が状態を管理、props で受け取る
   - 非制御コンポーネント: 自身で状態を管理

3. Phase 4 のカスタムフックを使用する
   - useUsers() → users, isLoading, error
   - useUserForm() → values, errors, handleChange, handleSubmit

4. 副作用（useEffect）の実装
   - データフェッチのトリガー
   - URLパラメータとの同期
   - 外部ライブラリの初期化
```

#### チェックリスト

```
□ 各 useState に適切な初期値が設定されている
□ 状態更新が不必要な再レンダリングを引き起こしていない
□ useEffect の依存配列が正しい
□ クリーンアップ関数が必要な場合、実装されている
□ 非同期処理のキャンセル / 競合状態の対処がある
```

### Step 6-3e: Props のバリデーション / デフォルト値設定

#### やること

- Props のデフォルト値を設定する
- 必要に応じてランタイムバリデーションを追加する

#### コーディング規約

```typescript
interface UserTableProps {
  users: User[]
  isLoading?: boolean
  emptyMessage?: string
  pageSize?: number
}

export function UserTable({
  users,
  isLoading = false,
  emptyMessage = 'データがありません',
  pageSize = 20,
}: UserTableProps) {
  // ...
}
```

#### チェックリスト

```
□ オプショナル props にデフォルト値がある
□ 必須 props が欠けている場合のエラーがわかりやすい（TypeScript で防止）
□ 不正な props の組み合わせがないか確認（相互排他的な props 等）
```

## Step 6-4: 中間コンポーネント（Container）の実装

### やること

- Layer 2以上のコンポーネントを実装する
- 子コンポーネントを組み合わせ、データフローを構築する

### 手順

```
1. 子コンポーネントを import する
2. 必要なカスタムフックを呼び出す
3. データの変換・フィルタリング（useMemo で最適化）
4. イベントハンドラを子コンポーネントに渡す
5. 状態パターン別のレンダリング分岐を実装する
```

### チェックリスト

```
□ 子コンポーネントへの props バケツリレーが過剰でないか
□ Context の導入が必要なレベルではないか（3階層以上の props 伝播）
□ 不要な再レンダリングが起きていないか（React DevTools で確認）
□ コンポーネントの責務が明確か（表示とロジックが混在していないか）
```

---

## Step 6-5: 状態パターン別の表示確認

### やること

- Phase 1 で定義した全状態パターンが正しく表示されることを確認する

### 確認手順

```
各状態について:
1. その状態を発生させる方法を確認する
2. 実際にその状態を再現する
3. 表示が仕様通りか確認する
4. 状態間の遷移が滑らかか確認する

必須確認パターン:
□ 初期ローディング → スケルトンUI / スピナー表示
□ データ取得成功（あり） → 正常表示
□ データ取得成功（なし） → エンプティステート
□ エラー → エラーメッセージ + リトライ
□ 再取得中 → 既存データ + ローディングインジケータ
□ フォームバリデーションエラー → エラー表示
□ 操作成功 → 成功フィードバック（トースト等）
```

### 完了条件

- 全状態パターンの表示が確認されている

---

## Step 6-6: コンポーネント間のデータフローの接続確認

### やること

- 全コンポーネントが正しくデータをやり取りしていることを確認する

### 確認項目

```
□ フィルター変更 → テーブルデータ更新
□ ソート変更 → テーブルデータ更新
□ ページネーション → テーブルデータ更新 + URLパラメータ更新
□ 新規作成 → モーダル → API → テーブル再取得
□ 削除 → 確認ダイアログ → API → テーブル再取得
□ エラー発生 → エラー表示コンポーネントへの伝播
```

### 完了条件

- 全データフローが正しく機能している
