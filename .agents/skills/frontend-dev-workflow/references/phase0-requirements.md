# Phase 0: 要件ヒアリング & 整理 — 詳細手順

このフェーズでは **一行もコードを書かない**。目的は曖昧さを完全に排除すること。

---

## Step 0-1: 画面の目的・ユーザーストーリーの確認

### やること

- この画面は **誰が・いつ・なぜ** 使うのかを明確にする
- ユーザーストーリー形式で記述する

### 質問テンプレート

```
この画面の主な利用者は誰ですか？（例：管理者、一般ユーザー、ゲスト）
この画面で達成したい主要なゴールは何ですか？
ユーザーはどの画面からこの画面に到達しますか？
この画面の後、ユーザーはどこに遷移しますか？
```

### 出力フォーマット

```
## ユーザーストーリー
- As a [ユーザー種別], I want to [アクション], so that [目的].
- As a [ユーザー種別], I want to [アクション], so that [目的].
```

### 完了条件

- ユーザーストーリーが最低1つ定義されている
- 利用者の種別が明確

---

## Step 0-2: 主要機能の洗い出し

### やること

- 画面に含まれるすべての機能を箇条書きで列挙する
- 各機能に **MUST / SHOULD / COULD** の優先度をつける

### 質問テンプレート

```
この画面で絶対に必要な機能は何ですか？
あると良いが、なくても最低限動く機能はありますか？
将来的に追加したいが、今回のスコープ外の機能はありますか？
```

### 出力フォーマット

```
## 機能一覧
| # | 機能名 | 説明 | 優先度 |
|---|--------|------|--------|
| 1 | ユーザー一覧表示 | ページネーション付きテーブル | MUST |
| 2 | 検索フィルター | 名前・メールで絞り込み | MUST |
| 3 | CSVエクスポート | 表示中のデータをCSV出力 | SHOULD |
```

### 完了条件

- 全機能に優先度がついている
- MUST 機能が最低1つ存在する

---

## Step 0-3: 画面遷移・ルーティングの確認

### やること

- この画面のURLパスを確定する
- 動的ルーティングの有無を確認する
- 関連画面との遷移関係を整理する

### 確認項目

```
1. この画面のURLパスは？ (例: /dashboard/users)
2. 動的セグメントはあるか？ (例: /users/[id])
3. クエリパラメータは使うか？ (例: /users?page=2&sort=name)
4. パラレルルート / インターセプトルートの必要性は？
5. 遷移元画面の一覧
6. 遷移先画面の一覧
```

### 出力フォーマット

```
## ルーティング
- パス: /dashboard/users
- 動的セグメント: なし
- クエリパラメータ: page, sort, search
- 遷移元: /dashboard (サイドバーリンク)
- 遷移先: /dashboard/users/[id] (行クリック)
```

### 完了条件

- URLパスが確定している
- 遷移元・遷移先が明記されている

---

## Step 0-4: データソースの確認

### やること

- 画面で表示・操作するデータの出どころを確認する
- 各データの取得方法（REST API / GraphQL / ローカルステート / 外部サービス）を明確にする

### 確認項目

```
1. メインデータのAPIエンドポイント（または取得方法）
2. リクエストメソッド (GET / POST / etc.)
3. リクエストパラメータ
4. レスポンス構造の概要（フィールド名だけでOK）
5. 書き込み操作の有無と対応するエンドポイント
6. リアルタイム更新の必要性（WebSocket / SSE / ポーリング）
7. 外部サービス連携（認証プロバイダ、ファイルストレージ等）
```

### 出力フォーマット

```
## データソース
| データ | エンドポイント | メソッド | 用途 |
|--------|--------------|---------|------|
| ユーザー一覧 | GET /api/users | GET | テーブル表示 |
| ユーザー作成 | POST /api/users | POST | 新規追加モーダル |
```

### 完了条件

- 全データソースが特定されている
- 取得方法が明記されている

---

## Step 0-5: 認証・認可の要件確認

### やること

- この画面にアクセスするための認証要件を確認する
- 画面内の要素に対するロールベースの表示制御を確認する

### 確認項目

```
1. 認証が必要か？ (public / authenticated / specific role)
2. 認証方式 (Cookie / JWT / OAuth)
3. ロール一覧と各ロールで見える要素の差分
4. 未認証時のリダイレクト先
5. 権限不足時の表示（非表示 / disabled / エラーメッセージ）
```

### 出力フォーマット

```
## 認証・認可
- 認証: 必要（JWT）
- 必要ロール: admin, manager
- ロール別制御:
  - admin: 全機能利用可能
  - manager: 削除ボタン非表示
  - 未認証: /login にリダイレクト
```

### 完了条件

- 認証要件が明記されている
- ロール別の表示制御が定義されている（該当する場合）

---

## Step 0-6: レスポンシブ対応の範囲確認

### やること

- 対応するブレークポイントと各サイズでのレイアウト差分を明確にする

### 確認項目

```
1. モバイル対応は必要か？
2. ブレークポイント定義 (sm / md / lg / xl)
3. モバイルで非表示にする要素はあるか？
4. モバイル専用のUIパターン（ボトムシート、ハンバーガーメニュー等）
5. タブレット特有のレイアウトはあるか？
```

### 出力フォーマット

```
## レスポンシブ対応
- モバイル (< 768px): テーブル → カードリスト表示、サイドバー → ハンバーガー
- タブレット (768px - 1024px): サイドバー折りたたみ
- デスクトップ (> 1024px): フルレイアウト
```

### 完了条件

- 対応範囲が明記されている
- サイズ別の差分が把握されている

---

## Step 0-7: 非機能要件

### やること

- パフォーマンス、アクセシビリティ、国際化に関する要件を確認する

### 確認項目

```
パフォーマンス:
1. 初期表示のパフォーマンス目標（LCP < 2.5s 等）
2. 大量データの取り扱い（仮想スクロール / ページネーション）
3. 画像最適化の必要性

アクセシビリティ:
4. WCAG準拠レベル (A / AA / AAA)
5. スクリーンリーダー対応の必須度
6. キーボードナビゲーション対応

国際化:
7. 多言語対応の必要性
8. RTL対応の必要性
9. 日付・数値のローカライズ
```

### 出力フォーマット

```
## 非機能要件
- パフォーマンス: ページネーション採用、画像はnext/image使用
- アクセシビリティ: WCAG 2.1 AA準拠
- i18n: 日本語のみ（今回スコープ）
```

### 完了条件

- 各カテゴリの要件が「対応する」「しない」で明記されている

---

## Step 0-8: 要件サマリーの作成とユーザー確認

### やること

- Step 0-1 〜 0-7 の結果を1つのサマリードキュメントに統合する
- スコープ内 / スコープ外を明確に分離する
- **ユーザーに提示して承認を得る**

### サマリーテンプレート

```markdown
# 要件サマリー: [画面名]

## 概要

[1-2文の画面説明]

## ユーザーストーリー

[Step 0-1 の結果]

## 機能一覧

[Step 0-2 の結果]

## ルーティング

[Step 0-3 の結果]

## データソース

[Step 0-4 の結果]

## 認証・認可

[Step 0-5 の結果]

## レスポンシブ対応

[Step 0-6 の結果]

## 非機能要件

[Step 0-7 の結果]

## スコープ外（今回実装しないもの）

- [明示的に除外する項目]

## 前提条件・依存

- [この実装が依存する外部要素]
```

### 完了条件

- サマリーが作成されている
- ユーザーが **明示的に承認** した（「OKです」「進めてください」等）
- スコープ外が明記されている

---

## Phase 0 完了チェック

Phase 0 を完了したら、以下をすべて確認:

```
✅ 要件サマリードキュメントが作成されている
✅ ユーザーが要件サマリーを承認した
✅ スコープ外の項目が明記されている
✅ ここまでコードは一切書いていない
```

すべてOKなら → **Phase 1 へ進む**
