# Phase 1: 画面仕様の確定 — 詳細手順

このフェーズでも **コードは書かない**。画面の構成要素をワイヤーフレームレベルで確定する。

---

## Step 1-1: 画面レイアウト構成の決定

### やること
- ページ全体のレイアウトパターンを決定する
- 既存の layout.tsx との関係を明確にする

### 確認項目
```
1. レイアウトパターンの選択:
   □ フルスクリーン（ヘッダーなし）
   □ ヘッダー + コンテンツ
   □ サイドバー + コンテンツ
   □ ヘッダー + サイドバー + コンテンツ
   □ ダッシュボード型（ヘッダー + サイドバー + メイン + サブパネル）
   □ カスタム

2. 既存 layout.tsx から引き継がれる要素は何か
3. この画面固有のレイアウト要素は何か
4. 固定要素（sticky header, fixed sidebar等）の有無
```

### 出力フォーマット
```
## レイアウト構成
┌─────────────────────────────────────┐
│            Header (共通layout)       │
├──────────┬──────────────────────────┤
│          │                          │
│ Sidebar  │     Main Content         │
│ (共通)    │     (この画面固有)         │
│          │                          │
├──────────┴──────────────────────────┤
│            Footer (なし)             │
└─────────────────────────────────────┘

- Header: 共通layout.tsx で定義済み
- Sidebar: 共通layout.tsx で定義済み
- Main Content: この画面で実装
```

### 完了条件
- レイアウトパターンが確定している
- 共通 layout vs 画面固有の境界が明確

---

## Step 1-2: セクション分割と各セクションの責務定義

### やること
- Main Content 領域を論理的なセクションに分割する
- 各セクションに名前、責務、含まれるUI要素を割り当てる

### 手順
1. 画面を上から下へスキャンし、視覚的なまとまりを特定する
2. 各まとまりに一意のセクション名をつける
3. 各セクションに含まれるUI要素を列挙する
4. セクション間の依存関係を特定する

### 出力フォーマット
```
## セクション構成

### Section A: ページヘッダー
- 責務: ページタイトルとアクションボタンの表示
- UI要素:
  - ページタイトル（h1）
  - 「新規作成」ボタン
  - パンくずリスト
- 依存: なし

### Section B: フィルター & 検索バー
- 責務: データの絞り込みUI
- UI要素:
  - テキスト検索入力
  - ステータスフィルター（ドロップダウン）
  - 日付範囲ピッカー
  - 「フィルタークリア」ボタン
- 依存: Section C（フィルター変更 → テーブル再描画）

### Section C: データテーブル
- 責務: データの一覧表示
- UI要素:
  - テーブルヘッダー（ソート可能）
  - テーブル行（クリックで詳細遷移）
  - ページネーション
  - 「選択中: N件」表示
  - 一括操作ドロップダウン
- 依存: Section B（フィルター条件を受け取る）
```

### 完了条件
- 全セクションに名前と責務がある
- 全UI要素がいずれかのセクションに割り当てられている
- セクション間の依存関係が明記されている

## Step 1-3: インタラクション一覧

### やること
- 画面上のすべてのユーザーインタラクションを列挙する
- 各インタラクションのトリガーと結果を定義する

### 出力フォーマット
```
## インタラクション一覧
| # | トリガー | 要素 | アクション | 結果 |
|---|---------|------|-----------|------|
| 1 | Click | 「新規作成」ボタン | モーダル表示 | 新規作成モーダルが開く |
| 2 | Input | 検索テキスト | デバウンス検索 | 300ms後にテーブル更新 |
| 3 | Click | テーブルヘッダー | ソートトグル | ASC/DESC切り替え、テーブル更新 |
| 4 | Click | テーブル行 | 画面遷移 | /users/[id] へ遷移 |
| 5 | Click | ページネーション | ページ切替 | テーブルデータ更新、URL更新 |
| 6 | Click | 削除ボタン | 確認ダイアログ | 確認 → API呼び出し → テーブル更新 |
| 7 | Submit | 新規作成フォーム | API POST | 成功 → モーダル閉じ → テーブル更新 |
| 8 | Hover | テーブル行 | ハイライト | 行背景色変更 |
| 9 | Keyboard | Escape | モーダル閉じ | 開いているモーダルを閉じる |
```

### 完了条件
- 全インタラクションがリストアップされている
- 各インタラクションに明確な結果が定義されている

---

## Step 1-4: 状態一覧の洗い出し

### やること
- 画面で発生しうるすべての状態パターンを列挙する
- 各状態での画面表示を定義する

### 標準状態パターン（最低限これらをチェック）
```
1. 初期ローディング: データ取得中の表示
2. 成功（データあり）: 正常表示
3. 成功（データなし / 空）: エンプティステート
4. エラー（ネットワーク）: 通信エラー表示
5. エラー（サーバー）: サーバーエラー表示
6. エラー（認証切れ）: セッション切れ表示
7. 部分ローディング: 一部のみ再取得中
8. 操作中（楽観的更新）: 削除中のアイテムのフェードアウト等
9. フォームバリデーションエラー: 入力エラー表示
10. 成功フィードバック: 操作成功後のトースト等
```

### 出力フォーマット
```
## 状態パターン
| # | 状態 | 表示内容 | 該当セクション |
|---|------|---------|--------------|
| 1 | 初期ローディング | スケルトンUI | Section C |
| 2 | 成功（データあり） | テーブル表示 | Section C |
| 3 | 成功（空） | 「データがありません」+ 作成リンク | Section C |
| 4 | ネットワークエラー | リトライボタン付きエラー表示 | Section C |
| 5 | 検索結果なし | 「条件に一致するデータがありません」 | Section C |
```

### 完了条件
- 上記の標準パターンが全てチェックされている
- 各状態の表示内容が定義されている

---

## Step 1-5: モーダル・ドロワー・トースト等のオーバーレイ要素の特定

### やること
- ページ上に重なって表示されるすべてのオーバーレイ要素を特定する
- 各オーバーレイのトリガー、内容、閉じ方を定義する

### 確認項目
```
□ モーダルダイアログ（作成 / 編集 / 確認 / 詳細表示）
□ ドロワー（サイドパネル）
□ トースト / スナックバー（成功 / エラー通知）
□ ツールチップ
□ ドロップダウンメニュー / ポップオーバー
□ コマンドパレット
□ ライトボックス / 画像プレビュー
```

### 出力フォーマット
```
## オーバーレイ要素
| # | 種別 | 名称 | トリガー | 内容 | 閉じ方 |
|---|------|------|---------|------|--------|
| 1 | Modal | 新規作成モーダル | 「新規作成」ボタン | フォーム3項目 | ×ボタン / ESC / 背景クリック |
| 2 | Dialog | 削除確認 | 削除アイコン | 確認メッセージ + ボタン2つ | キャンセル / 確認 |
| 3 | Toast | 操作成功 | CRUD完了後 | 「保存しました」等 | 3秒後自動消去 / ×ボタン |
```

### 完了条件
- 全オーバーレイ要素が特定されている
- 各要素のライフサイクル（開く → 中身 → 閉じる）が定義されている

---

## Step 1-6: 画面仕様書の作成

### やること
- Step 1-1 〜 1-5 の結果を統合した画面仕様書を作成する
- ユーザーに提示して承認を得る

### 仕様書テンプレート
```markdown
# 画面仕様書: [画面名]

## 1. レイアウト構成
[Step 1-1 の結果: ASCII図 + 説明]

## 2. セクション構成
[Step 1-2 の結果]

## 3. インタラクション一覧
[Step 1-3 の結果]

## 4. 状態パターン
[Step 1-4 の結果]

## 5. オーバーレイ要素
[Step 1-5 の結果]

## 6. 補足事項
[追加の注意点やデザイン上の制約]
```

### 完了条件
- 画面仕様書が作成されている
- ユーザーが **明示的に承認** した
