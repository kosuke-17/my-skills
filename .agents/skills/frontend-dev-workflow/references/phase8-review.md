# Phase 8: レビュー & リファクタ — 詳細手順

品質の最終確認と改善。全フェーズの成果物を横断的にチェックする。

---

## Step 8-1: コードレビューチェックリスト

### やること

- 全ファイルを対象に、以下のチェックリストを実行する

### 構造 & アーキテクチャ

```
□ feature ディレクトリ構成が Phase 2 のファイルマップと一致している
□ SC / CC の分類が適切（不必要に 'use client' をつけていない）
□ 循環依存がない
□ re-export (index.ts) が正しく構成されている
□ 共通コンポーネント vs feature 固有の切り分けが適切
□ コンポーネントの粒度が適切（大きすぎるコンポーネントがない）
```

### 型安全性

```
□ any 型が使われていない（やむを得ない場合は as でキャストし、コメントで理由を記載）
□ 型アサーション（as）の濫用がない
□ null / undefined の安全な取り扱い（optional chaining, nullish coalescing）
□ イベントハンドラの型が明示されている
□ ジェネリクスが適切に使われている
□ discriminated union が活用されている（状態パターン等）
```

### React パターン

```
□ key 属性が安定した値を使っている（index ではなく id）
□ useEffect の依存配列が正しい（ESLint ルールに従う）
□ useCallback / useMemo が適切に使われている（過剰使用も確認）
□ カスタムフックが再利用可能な形で設計されている
□ Provider のネストが深すぎない
□ forwardRef が必要なコンポーネントに適用されている
□ displayName が設定されている（デバッグ用）
```

### エラーハンドリング

```
□ API呼び出しに try/catch がある
□ エラー状態の UI が実装されている
□ エラーメッセージがユーザーフレンドリー
□ error.tsx がキャッチしきれないエラーがないか
□ フォームバリデーションエラーが適切に表示される
```

### 完了条件

- 全チェック項目が確認されている
- 問題があれば修正済み

## Step 8-2: TypeScript strict モードでのエラー確認

### やること

- TypeScript の strict モードでコンパイルエラーがないことを確認する

### 確認項目

```
□ tsc --noEmit でエラーがない
□ strict: true が tsconfig.json で有効
□ strictNullChecks による null チェック漏れがない
□ strictFunctionTypes によるコールバック型の不一致がない
□ noImplicitAny で暗黙の any がない
```

### 完了条件

- TypeScript コンパイルエラーが0件

## Step 8-3: アクセシビリティ監査

### やること

- セマンティックHTML、ARIA属性、キーボード操作を確認する

### セマンティックHTML

```
□ 適切なランドマーク要素（header, nav, main, aside, footer）
□ 見出しレベルの順序（h1 → h2 → h3、スキップなし）
□ リスト要素の適切な使用（ul/ol/dl）
□ テーブルの適切な構造（thead, tbody, th[scope]）
□ フォーム要素に label が関連付けられている
□ ボタンとリンクの使い分け（操作 = button、遷移 = a/Link）
```

### ARIA 属性

```
□ 動的コンテンツに aria-live が設定されている
□ モーダルに aria-modal="true" と role="dialog" がある
□ アイコンのみのボタンに aria-label がある
□ ローディング状態に aria-busy がある
□ 展開/折りたたみに aria-expanded がある
□ タブUIに適切な role (tablist, tab, tabpanel) がある
□ 装飾的な画像に aria-hidden="true" がある
```

### キーボード操作

```
□ Tab キーで全インタラクティブ要素にフォーカスできる
□ フォーカス順序が視覚的な順序と一致する
□ Enter / Space でボタンが動作する
□ Escape でモーダル / ドロップダウンが閉じる
□ 矢印キーでリスト / タブを移動できる
□ フォーカストラップがモーダル内で機能する
□ フォーカスインジケータ（:focus-visible）が見える
```

### カラーコントラスト

```
□ テキストのコントラスト比が 4.5:1 以上（WCAG AA）
□ 大きなテキスト（18px以上 or 14px bold以上）は 3:1 以上
□ フォーカスインジケータのコントラストが 3:1 以上
□ 色だけで情報を伝えていない（エラーは色 + アイコン/テキスト）
```

### 完了条件

- 全チェック項目が確認されている
- 問題があれば修正済み

## Step 8-4: パフォーマンス確認

### やること

- 不要な再レンダリングやパフォーマンス上の問題がないか確認する

### 再レンダリング確認

```
□ React DevTools Profiler で不要な再レンダリングを特定
□ コールバック props による不要な再レンダリングがない（useCallback）
□ オブジェクト / 配列の inline 生成による再レンダリングがない（useMemo）
□ Context の値変更による広範囲の再レンダリングがない
□ 状態のリフトアップが適切（不必要に高い位置で状態を持っていない）
```

### バンドルサイズ

```
□ 不要な import がない
□ dynamic import（next/dynamic）の活用（大きなコンポーネント）
□ ツリーシェイキングが効く形の import（名前付き import）
□ 大きなライブラリの部分 import
```

### データ取得

```
□ N+1 問題がない（一覧取得 → 各行で追加フェッチ等）
□ 不必要なデータ取得がない
□ データのキャッシュが適切
□ 画像の最適化（next/image, width/height指定, lazy loading）
```

### 完了条件

- パフォーマンス上の明らかな問題がない

## Step 8-5: 命名規則・コード規約の統一確認

### やること

- プロジェクト全体で命名規則とコード規約が統一されていることを確認する

### 命名規則

```
□ コンポーネント: PascalCase (UserTable)
□ ファイル名: kebab-case (user-table.tsx)
□ フック: camelCase + use プレフィックス (useUsers)
□ 型: PascalCase (UserTableProps)
□ 定数: UPPER_SNAKE_CASE (MAX_PAGE_SIZE)
□ 変数/関数: camelCase (fetchUsers)
□ CSSクラス: プロジェクト規約に従う（kebab-case / BEM / Tailwind）
□ ディレクトリ: kebab-case (user-management)
```

### コード規約

```
□ import 順序の統一（React → Next.js → 外部ライブラリ → 内部モジュール → 型）
□ export スタイルの統一（named export が基本、page.tsx のみ default export）
□ コンポーネント定義スタイルの統一（function 宣言 or アロー関数）
□ props の destructuring スタイルの統一
□ コメントの言語・スタイルの統一
□ console.log の残存がない
```

### 完了条件

- 命名規則が統一されている
- コード規約に違反する箇所がない

## Step 8-6: TODO / FIXME の棚卸し

### やること

- コード内の TODO / FIXME / HACK コメントを確認する
- 解決すべきものと、将来対応でよいものを分類する

### 手順

```
1. grep -r "TODO\|FIXME\|HACK\|XXX" で全件抽出
2. 各項目を以下に分類:
   - 今回修正する → 修正する
   - 将来対応（Issue化） → Issue番号を付与してコメントに追記
   - 不要（すでに解決済み） → コメント削除
```

### 完了条件

- 全 TODO/FIXME が分類されている
- 今回修正すべきものは修正済み
- 将来対応のものは Issue 番号が付与されている
