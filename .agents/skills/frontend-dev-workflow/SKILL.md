# Frontend Dev Workflow

フロントエンド実装を **9つのフェーズ × 合計40+ステップ** に分解し、各ステップの完了条件を満たしてから次へ進む段階的ワークフロー。

## 前提技術スタック

- Next.js App Router (App Directory)
- TypeScript (strict mode)
- Feature-based directory structure
- React Server Components / Client Components の明示的分離

> ユーザーが異なるスタックを指定した場合はそちらに合わせつつ、フェーズ構造は維持する。

---

## ワークフロー全体像

```
Phase 0: 要件ヒアリング & 整理
Phase 1: 画面仕様の確定
Phase 2: ディレクトリ設計 & ファイルマップ
Phase 3: 型定義 (types)
Phase 4: API層 / データ取得層 (api / hooks)
Phase 5: 共通コンポーネント / UI部品
Phase 6: Feature コンポーネント実装
Phase 7: page.tsx 組み立て
Phase 8: レビュー & リファクタ
```

各フェーズの詳細手順は `references/` に格納。SKILL.md ではフェーズ概要とゲート条件（次へ進む判定基準）を定義する。

---

## フェーズ制御ルール

1. **必ずPhase 0から開始する。** ユーザーが「もう要件は決まっている」と言っても、確認のためにPhase 0の質問を最低限実施する。
2. **各フェーズの末尾にあるゲート条件をすべて満たしてから次フェーズへ進む。**
3. **ユーザーに各フェーズの完了を明示的に伝え、次フェーズへ進む許可を取る。**
4. **コードを書くのはPhase 3以降のみ。** Phase 0〜2は設計ドキュメントとテキストのみ。
5. **途中フェーズからの開始を要求された場合は、** 先行フェーズの成果物（型定義、ディレクトリ構成など）を確認してから着手する。

---

## Phase 0: 要件ヒアリング & 整理

**目的**: 曖昧さを排除し、実装スコープを確定する

詳細手順 → `references/phase0-requirements.md` を読むこと

### ステップ概要

0-1. 画面の目的・ユーザーストーリーの確認
0-2. 主要機能の洗い出し（箇条書き）
0-3. 画面遷移・ルーティングの確認
0-4. データソースの確認（API / ローカル / 外部）
0-5. 認証・認可の要件確認
0-6. レスポンシブ対応の範囲確認
0-7. 非機能要件（パフォーマンス、アクセシビリティ、i18n）
0-8. 要件サマリーの作成とユーザー確認

### ゲート条件

- [ ] 要件サマリードキュメントが作成されている
- [ ] ユーザーが要件サマリーを承認した
- [ ] スコープ外の項目が明記されている

---

## Phase 1: 画面仕様の確定

**目的**: ワイヤーフレームレベルで画面の構成要素を確定する

詳細手順 → `references/phase1-screen-spec.md` を読むこと

### ステップ概要

1-1. 画面レイアウト構成の決定（ヘッダー/サイド/メイン/フッター）
1-2. セクション分割と各セクションの責務定義
1-3. インタラクション一覧（クリック、入力、ホバー、スクロール等）
1-4. 状態一覧の洗い出し（ローディング、エラー、空、成功、部分表示）
1-5. モーダル・ドロワー・トースト等のオーバーレイ要素の特定
1-6. 画面仕様書の作成

### ゲート条件

- [ ] 全セクションの責務と含まれるUI要素が文書化されている
- [ ] インタラクション一覧が完成している
- [ ] 全状態パターンが列挙されている
- [ ] ユーザーが画面仕様を承認した

---

## Phase 2: ディレクトリ設計 & ファイルマップ

**目的**: 実装前にすべてのファイルパスと役割を確定する

詳細手順 → `references/phase2-directory.md` を読むこと

### ステップ概要

2-1. feature ディレクトリ名の決定
2-2. ファイルツリーの作成（全ファイルパス + 1行説明）
2-3. Server Component / Client Component の分類
2-4. 共通コンポーネント vs feature固有コンポーネントの切り分け
2-5. re-export 用 index.ts の設計
2-6. ファイルマップのユーザー確認

### ゲート条件

- [ ] 完全なファイルツリーが作成されている
- [ ] 各ファイルに SC/CC の分類がついている
- [ ] ユーザーがファイルマップを承認した

---

## Phase 3: 型定義 (types)

**目的**: コードの最初の一歩として型定義を確定する

詳細手順 → `references/phase3-types.md` を読むこと

### ステップ概要

3-1. APIレスポンス型の定義
3-2. フロントエンド用ドメインモデル型の定義
3-3. コンポーネントProps型の定義
3-4. フォーム入力値・バリデーション型の定義
3-5. 状態管理用の型（State / Action）定義
3-6. 型ファイルの作成と export 構成

### ゲート条件

- [ ] 全APIレスポンスに対応する型が定義されている
- [ ] 全コンポーネントのProps型が定義されている
- [ ] 型ファイルが作成され、コンパイルエラーがない

---

## Phase 4: API層 / データ取得層

**目的**: データの入出力インターフェースを実装する

詳細手順 → `references/phase4-api.md` を読むこと

### ステップ概要

4-1. API クライアント関数の実装
4-2. Server Actions の実装（必要な場合）
4-3. カスタムフックの実装（useXxx）
4-4. エラーハンドリングパターンの統一
4-5. ローディング・キャッシュ戦略の決定
4-6. API層の単体テスト（モック含む）

### ゲート条件

- [ ] 全APIエンドポイントに対応する関数/Actionがある
- [ ] エラーハンドリングが統一されている
- [ ] 型安全性が確保されている

---

## Phase 5: 共通コンポーネント / UI部品

**目的**: feature内外で再利用されるUI部品を先に実装する

詳細手順 → `references/phase5-ui-components.md` を読むこと

### ステップ概要

5-1. 既存の共通コンポーネントライブラリの確認
5-2. 新規共通コンポーネントの洗い出し
5-3. 各コンポーネントのインターフェース設計（Props）
5-4. コンポーネント実装（見た目 + 基本動作）
5-5. バリアント・サイズ・状態の網羅
5-6. Storybook / プレビュー用の確認

### ゲート条件

- [ ] Phase 6 で必要な共通コンポーネントがすべて実装済み
- [ ] Props が Phase 3 の型定義と整合している

---

## Phase 6: Feature コンポーネント実装

**目的**: feature ディレクトリ内の各コンポーネントを実装する

詳細手順 → `references/phase6-feature-components.md` を読むこと

### ステップ概要

6-1. feature コンポーネントの実装順序の決定（依存関係順）
6-2. 末端コンポーネント（Leaf）から実装開始
6-3. 各コンポーネントの実装

- 6-3a. JSX構造の実装
- 6-3b. スタイリングの適用
- 6-3c. イベントハンドラの接続
- 6-3d. 状態管理ロジックの組み込み
- 6-3e. Props のバリデーション / デフォルト値設定
  6-4. 中間コンポーネント（Container）の実装
  6-5. 状態パターン別の表示確認（ローディング/エラー/空/正常）
  6-6. コンポーネント間のデータフローの接続確認

### ゲート条件

- [ ] 全feature コンポーネントが実装されている
- [ ] 全状態パターンが表示可能
- [ ] Phase 4 のフックと正しく接続されている

---

## Phase 7: page.tsx 組み立て

**目的**: ルーティングに対応するページファイルを完成させる

詳細手順 → `references/phase7-page.md` を読むこと

### ステップ概要

7-1. page.tsx のファイル作成とメタデータ設定
7-2. layout.tsx の確認・調整
7-3. Server Component としてのデータ取得
7-4. Client Component への Props 受け渡し
7-5. Suspense / loading.tsx / error.tsx の設定
7-6. ページレベルのレイアウト組み立て
7-7. 画面遷移（Link / useRouter）の接続
7-8. SEO 対応（metadata export / generateMetadata）
7-9. 全体結合テスト

### ゲート条件

- [ ] page.tsx が正常にレンダリングされる
- [ ] 全状態パターン（loading/error/empty/success）が動作する
- [ ] 画面遷移が正しく機能する

---

## Phase 8: レビュー & リファクタ

**目的**: 品質の最終確認と改善

詳細手順 → `references/phase8-review.md` を読むこと

### ステップ概要

8-1. コードレビューチェックリストの実行
8-2. TypeScript strict モードでのエラー確認
8-3. アクセシビリティ監査（セマンティックHTML、ARIA、キーボード操作）
8-4. パフォーマンス確認（不要な再レンダリング、バンドルサイズ）
8-5. 命名規則・コード規約の統一確認
8-6. TODO / FIXME の棚卸し
8-7. 最終成果物の提示

### ゲート条件

- [ ] 全チェックリスト項目がPass
- [ ] ユーザーに最終成果物が提示されている

---

## 部分実行モード

ユーザーが特定のフェーズだけを要求した場合（例：「型定義だけ作って」）、以下のルールで対応する:

1. 要求されたフェーズの **先行フェーズの成果物が存在するか確認する**
2. 存在しない場合、最低限必要な情報をヒアリングで補完する
3. 要求されたフェーズのみを実行し、後続フェーズには進まない
4. 実行したフェーズの成果物を明示的にまとめて提示する
