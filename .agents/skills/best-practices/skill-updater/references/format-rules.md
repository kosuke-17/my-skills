# フォーマット整合性ルール

**原則**: 更新されたコンテンツが既存のリファレンスファイルと構造的・視覚的に一致していることを保証する

## リファレンスファイルの標準構造

リファレンスファイルは以下のセクション順序に従う。すべてのセクションが必須ではないが、存在する場合はこの順序を維持する：

```
# [トピック]のベストプラクティス      ← H1: タイトル
**原則**: ...                          ← 太字の原則ステートメント
[導入文]                               ← 1〜2文の説明
## チェック観点                        ← チェック項目のリスト
## 違反例                              ← ❌ 悪い例のコードブロック
## 改善例                              ← ✅ 良い例のコードブロック
## よくある違反パターン                 ← パターン形式（悪い例/良い例ペア）
## コードレビュー時のチェックポイント    ← 番号付きリスト
## 注意点                              ← 箇条書き
```

## フォーマットルール

### 1. コードブロック

- 言語タグは対象スキルの主要言語に合わせる
  - Rails スキル → `ruby`
  - TypeScript スキル → `typescript`
  - その他 → 対象ファイルの既存コードブロックに従う
- コードブロック内のコメントで ❌ / ✅ マーカーを使用する

```markdown
\```ruby
# ❌ 悪い例: [簡潔な説明]
[コード]
\```

\```ruby
# ✅ 良い例: [簡潔な説明]
[コード]
\```
```

### 2. 見出しレベル

- `#` (H1): ファイルタイトルのみ（1ファイルに1つ）
- `##` (H2): メインセクション
- `###` (H3): 例やパターンのサブセクション

### 3. リスト形式

**チェック観点:**
```markdown
- [疑問形または体言止めで記述]
```

**チェックポイント:**
```markdown
1. **[太字キーワード]**: [説明文]
```

**注意点:**
```markdown
- **[太字キーワード]**: [説明文]
```

### 4. 追記位置

- 各セクションの **末尾** に追加する
- セクション内の途中に挿入しない
- セクション間の空行は既存ファイルに合わせる（通常1行）

### 5. 番号の連番

- `### パターンN` や `### 例N` の番号は、既存の最後の番号に続ける
- 番号を飛ばさない
- 既存の番号を変更しない

## 更新前の確認事項

更新を実行する前に、以下を確認する：

1. **対象ファイルを読む**: 更新対象のリファレンスファイル全体を読み、現在の構造を把握する
2. **セクション名を確認**: 対象スキルのセクション名が標準構造と異なる場合は、既存のセクション名に合わせる（標準構造を強制しない）
3. **言語タグを確認**: 既存のコードブロックで使用されている言語タグを確認し、同じものを使用する
4. **番号を確認**: 既存のパターン・例・チェックポイントの最後の番号を確認する
5. **トーンを確認**: 既存の記述スタイル（敬体/常体、日本語/英語）に合わせる
