# 更新判定基準

**原則**: レビュー中に発見されたパターンが、スキルの永続的な改善につながる場合のみ更新する

## 追加すべきパターンの条件

以下の5つの基準のうち、**3つ以上を満たす**場合に追加候補として採用する：

### 1. 汎用性
- 複数のプロジェクトやコードベースで適用可能なパターンか
- 特定のプロジェクト構成に依存しないか

### 2. 再現性
- 今後のレビューでも繰り返し登場する可能性が高いか
- 一般的な開発者が陥りやすいパターンか

### 3. 重要度
- 見逃した場合にバグ・セキュリティ問題・パフォーマンス劣化につながるか
- コードの保守性に大きな影響を与えるか

### 4. 表現可能性
- 悪い例（❌）と良い例（✅）のペアで明確に表現できるか
- 簡潔な説明で理解できるか

### 5. 独自性
- 既存リファレンスで（暗黙的にも）カバーされていないか
- 既存のチェックポイントや違反パターンと重複しないか

## 追加すべきでないパターン

以下のいずれかに該当する場合は **追加しない**：

- **一度限り**: 特殊なコードベース構成でのみ発生する、再現性のないケース
- **主観的**: 客観的な品質改善ではなく、個人のスタイルの好みに基づくもの
- **矛盾**: 既存のベストプラクティスと矛盾するもの（既存を変更すべき場合はユーザーに相談）
- **自明**: ドキュメント化する価値がないほど当たり前のこと（例: 「構文エラーを残さない」）
- **バージョン限定**: 非推奨バージョンやニッチなフレームワーク版でのみ適用されるもの

## プロジェクト固有ルールの扱い

プロジェクト固有の規約（例: 「主キーは必ずUUIDを使用」「APIレスポンスは必ずSerializerを経由」）も追加可能。

ただし、以下のルールに従う：

1. リファレンスファイル内で `<!-- project-specific -->` HTMLコメントを付与してマーキングする
2. 該当パターンの説明に「プロジェクト固有」である旨を明記する
3. 汎用的なベストプラクティスとは区別できるようにする

```markdown
<!-- project-specific -->
### パターンN: UUIDを主キーとして使用

このプロジェクトでは、すべてのテーブルで主キーにUUIDを使用する規約がある。

❌ 悪い例:
\```ruby
create_table :users do |t|
  t.timestamps
end
\```

✅ 良い例:
\```ruby
create_table :users, id: :uuid do |t|
  t.timestamps
end
\```
```

## 判定フローチャート

```
発見したパターン
  ↓
[既存リファレンスにカバー済み?] → Yes → スキップ
  ↓ No
[追加すべきでないパターンに該当?] → Yes → スキップ（理由を記録）
  ↓ No
[5つの基準のうち3つ以上を満たす?] → No → スキップ（理由を記録）
  ↓ Yes
[プロジェクト固有か?] → Yes → project-specificマーキング付きで追加
  ↓ No
汎用パターンとして追加
```
