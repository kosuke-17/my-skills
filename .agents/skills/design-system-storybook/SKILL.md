# Design System & Storybook

デザインシステムの構築を **5つのフェーズ** に分解し、デザイントークン設計からコンポーネント実装・Storybookカタログ整備・テストまでを段階的に進めるワークフロー。

## 前提技術スタック

- React + TypeScript (strict mode)
- Tailwind CSS または CSS変数 (どちらにも対応)
- Storybook v8 (CSF3形式)
- Vite または Next.js

> ユーザーが異なるスタックを指定した場合はそちらに合わせつつ、フェーズ構造は維持する。

---

## ワークフロー全体像

```
Phase 0: 現状把握 & 要件定義
Phase 1: デザイントークン設計
Phase 2: コンポーネントアーキテクチャ設計
Phase 3: Storybook セットアップ
Phase 4: コンポーネント実装 & Story 作成
Phase 5: テスト・ドキュメント・公開
```

各フェーズの詳細手順は `references/` に格納。SKILL.md ではフェーズ概要とゲート条件を定義する。

---

## フェーズ制御ルール

1. **必ず Phase 0 から開始する。** 既存プロジェクトへの導入か新規構築かを最初に確認する。
2. **各フェーズの末尾にあるゲート条件をすべて満たしてから次フェーズへ進む。**
3. **ユーザーに各フェーズの完了を伝え、次フェーズへ進む許可を取る。**
4. **コードを書くのは Phase 2 以降のみ。** Phase 0〜1 は設計ドキュメントとトークン定義のみ。
5. **部分実行モードに対応する。** 「Storybookだけ設定したい」「トークンだけ整理したい」などの場合は該当フェーズのみ実行する。

---

## Phase 0: 現状把握 & 要件定義

**目的**: プロジェクトの現状とデザインシステムの対象スコープを確定する

詳細手順 → `references/phase0-audit.md` を読むこと

### ステップ概要

0-1. プロジェクトの技術スタック確認
0-2. 既存UIライブラリ・デザインシステムの有無の棚卸し
0-3. デザインツール連携の確認（Figma / Sketch / XD 等）
0-4. 対象スコープの確定（新規構築 / 既存拡張 / Storybook導入のみ）
0-5. コンポーネント一覧の初期リスト作成
0-6. 要件サマリーのユーザー確認

### ゲート条件

- [ ] 技術スタックが確定している
- [ ] 対象スコープ（新規 or 既存拡張）が明確になっている
- [ ] 作成・整備するコンポーネントの初期一覧がある
- [ ] ユーザーが要件サマリーを承認した

---

## Phase 1: デザイントークン設計

**目的**: デザインの最小単位となるトークンを定義し、一貫性の基盤を作る

詳細手順 → `references/phase1-design-tokens.md` を読むこと

### ステップ概要

1-1. カラートークン（プリミティブ → セマンティック）の設計
1-2. タイポグラフィトークン（font-size / weight / line-height / family）の設計
1-3. スペーシングスケールの設計（4px / 8px ベース）
1-4. シャドウ・エレベーション・ボーダーRadiusの定義
1-5. ブレークポイント・アニメーション/モーショントークンの定義
1-6. 実装形式の決定（CSS変数 / JS定数 / 両方）
1-7. ダークモード対応有無の確認
1-8. トークン定義ファイルの作成

### ゲート条件

- [ ] 全カテゴリのトークンが定義されている
- [ ] プリミティブとセマンティックの二層構造になっている
- [ ] トークン定義ファイルが作成されコンパイルエラーがない
- [ ] ユーザーがトークン設計を承認した

---

## Phase 2: コンポーネントアーキテクチャ設計

**目的**: コンポーネントの設計方針・命名規則・ディレクトリ構成を確定する

詳細手順 → `references/phase2-component-arch.md` を読むこと

### ステップ概要

2-1. Atomic Design 分類の適用（atoms / molecules / organisms）
2-2. コンポーネント API 設計原則の定義
2-3. バリアントシステムの設計（CVA / tailwind-variants 等）
2-4. ディレクトリ構成テンプレートの決定
2-5. コンポーネントのファイル構成ルールの確定
2-6. 命名規則の統一
2-7. ファイルマップのユーザー確認

### ゲート条件

- [ ] Atomic Design の分類基準が定義されている
- [ ] バリアントシステムの実装方針が決まっている
- [ ] ディレクトリ構成テンプレートが作成されている
- [ ] ユーザーがアーキテクチャ設計を承認した

---

## Phase 3: Storybook セットアップ

**目的**: コンポーネントカタログのインフラを整備する

詳細手順 → `references/phase3-storybook-setup.md` を読むこと

### ステップ概要

3-1. Storybook のインストール・初期化
3-2. 推奨アドオンの追加（a11y, docs, controls, viewport, backgrounds）
3-3. `.storybook/main.ts` の設定（ストーリーパス、フレームワーク、アドオン）
3-4. `.storybook/preview.ts` の設定（グローバルデコレーター、パラメーター）
3-5. デザイントークンの Storybook への反映（グローバルスタイル）
3-6. 動作確認（`pnpm storybook` で起動確認）

### ゲート条件

- [ ] Storybook が起動する
- [ ] 推奨アドオンが有効になっている
- [ ] デザイントークンが Storybook に反映されている
- [ ] TypeScript エラーがない

---

## Phase 4: コンポーネント実装 & Story 作成

**目的**: コンポーネントを実装し、同時に Storybook のカタログを整備する

詳細手順 → `references/phase4-story-writing.md` を読むこと

### ステップ概要

4-1. 実装順序の決定（依存関係の少ない atoms から開始）
4-2. 各コンポーネントの実装

- 4-2a. Props 型定義（Phase 1 トークンを参照）
- 4-2b. バリアント定義（CVA 等）
- 4-2c. JSX 実装とスタイリング
- 4-2d. アクセシビリティ対応（aria属性、キーボード操作）

4-3. 各コンポーネントの Story 作成（CSF3形式）

- 4-3a. meta（default export）の設定
- 4-3b. 全バリアント・状態のストーリー定義
- 4-3c. ArgTypes と Controls の設定
- 4-3d. play 関数によるインタラクションテスト（必要な場合）

4-4. Storybook 上での全バリアント・状態の目視確認

### ゲート条件

- [ ] 全コンポーネントが実装されている
- [ ] 各コンポーネントに Story が存在する
- [ ] 全バリアント・全状態のストーリーが揃っている
- [ ] Storybook 上でエラーなく表示される

---

## Phase 5: テスト・ドキュメント・公開

**目的**: 品質保証・ドキュメント整備・公開フローを整える

詳細手順 → `references/phase5-testing-docs.md` を読むこと

### ステップ概要

5-1. アクセシビリティテストの実行（@storybook/addon-a11y）
5-2. インタラクションテストの実行（play 関数 + @storybook/test）
5-3. ビジュアルリグレッションテストの設定（Chromatic）
5-4. MDX によるコンポーネントドキュメントの整備
5-5. Storybook のビルドと公開設定
5-6. リリースフロー・Changelog の整備（必要な場合）

### ゲート条件

- [ ] a11y テストがパスしている（または既知の例外が文書化されている）
- [ ] `storybook build` が成功する
- [ ] 主要コンポーネントにドキュメントが整備されている

---

## 部分実行モード

ユーザーが特定のフェーズだけを要求した場合:

| 要求 | 開始フェーズ | 前提確認事項 |
|------|------------|------------|
| 「Storybookだけ設定したい」 | Phase 3 | 技術スタック確認のみ実施 |
| 「デザイントークンだけ整理したい」 | Phase 1 | 既存トークン有無を確認 |
| 「Storyを書きたい」 | Phase 4 | Storybook が設定済みか確認 |
| 「ビジュアルテストを導入したい」 | Phase 5 | 既存Story数を確認 |
