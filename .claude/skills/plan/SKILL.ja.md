# CDD実装プランニング

機能またはコンポーネントの説明を受け取り、CDDベースの実装プランを作成する。

## ワークフロー

### ステップ1: 要件分析

$ARGUMENTS を分析し、以下を明確にする:

- 機能/UIの概要
- ユーザーインタラクション
- 必要なデータ構造

不明確な点がある場合はAskUserQuestionで確認する。

### ステップ2: 既存コンポーネント調査

Storybook MCPとファイルシステムを使って既存コンポーネントを調査する。

**調査方法（Storybook MCPが利用可能な場合は優先）:**

1. **Storybook MCP**（主要）: `mcp__storybook-mcp__list-all-documentation` を `withStoryIds: true` で呼び出し、ストーリーID付きの完全なコンポーネントリストを取得
2. **Storybook MCP**（詳細）: 候補コンポーネントごとに `mcp__storybook-mcp__get-documentation` を呼び出し、props、バリアント、使用パターンを確認
3. **ファイルシステム**（補助）: ストーリーのないコンポーネントを `components/`、`features/`、`hooks/` で確認

**調査対象:**

- `components/` — 共有UIコンポーネント
- `features/` — 機能固有のコンポーネント
- `hooks/` — 共有カスタムフック

再利用を優先し、新規作成を最小化する（DRY原則）。新しいコンポーネントAPIを設計する前に、Storybookドキュメントで既存コンポーネントのAPIを理解する。

### ステップ3: Atomic Designコンポーネント分解

以下の階層に分解する:

| レベル | 説明 | 例 |
| --------- | --------------------------------- | ---------------------------------- |
| Atoms | 最小のUI要素 | Button, Input, Badge |
| Molecules | Atomsの組み合わせ | SearchBar, FormField |
| Organisms | 複雑なUIセクション | WorkFilter, ProfileCard |
| Pages | ページ全体の構成（ストーリーなし） | WorkListPage |

**基準:**

- 単一責任の原則 — 各コンポーネントは一つのことだけを行う
- 既存コンポーネントとの重複を避ける
- コンポーネントの粒度をデータモデルに合わせる

### ステップ4: プロパティ設計

各コンポーネントのpropsインターフェースを設計する。

**規約:**

- booleanプロパティは `is`、`has`、`can` プレフィックスを使用
- コールバックプロパティは `on` プレフィックスを使用
- 外部スタイル拡張用に `className` プロパティを含める
- CVAで管理するバリアントを特定する

### ステップ5: 実装順序の決定

ボトムアップの実装順序を決定する:

```
1. AtomA (静的UI) -> ストーリー -> テスト
2. AtomB (静的UI) -> ストーリー -> テスト
3. MoleculeA (静的UI) -> ストーリー -> テスト
4. 動的UI追加（state、イベント） -> play関数テスト
5. OrganismA -> ストーリー -> テスト
6. PageA（ストーリー/テスト不要）
```

### ステップ6: テスト・ストーリー計画

各コンポーネントのストーリーとテスト戦略を決定する:

| テストタイプ | 対象 | ツール |
| ------------------ | -------------------- | ---------------------- |
| ユニットテスト | 全コンポーネント | Jest + Testing Library |
| インタラクションテスト | 動的コンポーネント | Storybook play関数 |
| VRT | 全ストーリー | Chromatic |

### ステップ7: プランファイルの出力

以下の形式でプランファイルを出力する:

- **出力先**: `.claude/plans/`
- **ファイル名**: `YYYY-MM-DD-HHMMSS-{識別子}.plan.md`

---

## プランファイルテンプレート

以下の構造でプランファイルを出力する:

**見出し**: `# 実装プラン: {機能名}`

**セクション構成:**

1. **概要** — 機能の説明
2. **コンポーネント分解** — Atoms / Molecules / Organisms / Pages ごとにコンポーネント名、新規/既存、責務をリスト
3. **プロパティ設計** — 各コンポーネントのTypeScript型定義
4. **実装順序** — 各コンポーネントのステップごとにチェックボックス付き番号リスト:
   - `[ ] 静的UI実装 (index.tsx)`
   - `[ ] ストーリー作成 (index.stories.tsx)`
   - `[ ] ユニットテスト (index.test.tsx)`
   - `[ ] 動的UI追加 (hooks.ts)` — 必要な場合
   - `[ ] インタラクションテスト (play関数)` — 必要な場合
5. **Storybook計画** — ストーリー名とplay関数の有無のテーブル
6. **テスト計画** — テスト観点のテーブル
7. **カスタムフック計画** — フック名、責務、配置場所
8. **備考** — 追加の考慮事項

---

## プランチェックリスト

出力前に確認:

- [ ] Atomic Design階層で分類されている
- [ ] 既存コンポーネントの再利用が検討されている
- [ ] 実装順序がボトムアップである
- [ ] 各コンポーネントのプロパティが設計されている
- [ ] Storybookストーリーが計画されている（Pages除く）
- [ ] テスト計画が含まれている
- [ ] カスタムフックの抽出が計画されている（必要な場合）
